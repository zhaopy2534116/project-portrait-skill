# 02 - 自评估体系设计

本文档设计一套用于持续优化 `project-portrait` skill 的自评估升级体系。目标是通过“样例项目 -> 调用 skill -> 评估 -> 失败归因 -> 修改 skill -> 回归验证”的循环，让 skill 的通用性、严谨性和文档质量持续提升。

## 1. 设计目标

这套体系要解决的问题不是“让模型自己随意优化自己”，而是建立一个可控、可复现、可比较的实验系统。

每次升级都应该能回答：

- 为什么要改？
- 改了哪个规则、模板、参考文档或脚本？
- 哪些样例项目变好了？
- 有没有引入回归？
- 这次改动是否值得合并？

## 2. 可借鉴的思路

### 2.1 OpenAI Evals

借鉴点：把能力拆成可重复运行的 eval case，并用结构化结果比较不同版本。

在本项目中对应为：

- 每个样例项目是一个 eval case。
- 每次运行 skill 都生成独立 run 目录。
- 每次修改 skill 后，对比新旧评分和失败类型。

### 2.2 LangSmith Evaluation

借鉴点：围绕数据集、评估器、反馈和版本对比构建 LLM 应用评估流程。

在本项目中对应为：

- 样例项目集就是数据集。
- 规则检查、LLM-as-judge、人工抽检都是评估器。
- failure report 和 reflection 是反馈。
- skill 的 git commit 或版本号是被比较对象。

### 2.3 DeepEval

借鉴点：将确定性指标和 LLM judge 指标结合，避免所有判断都依赖模型主观评分。

在本项目中对应为：

- 断链、空标题、占位符、文件引用不存在等用脚本检查。
- 类型适配、可读性、信息密度等用 LLM judge 辅助评估。
- blocking 问题优先由确定性规则定义。

### 2.4 Self-Refine

借鉴点：生成 -> 反馈 -> 改进的迭代闭环。

在本项目中对应为：

- 先用 skill 生成项目画像文档。
- 再对文档进行质量评估。
- 根据评估结果提出最小修改建议。
- 修改 skill 后重新运行同一 case 验证。

### 2.5 Reflexion

借鉴点：把失败经验沉淀为可复用的反思记录，而不是只看最终分数。

在本项目中对应为：

- 每次 eval run 生成 `reflection.md`。
- 记录失败现象、失败原因、对应规则缺口、建议修改位置。
- 后续修改 skill 时优先参考历史反思。

### 2.6 SWE-bench

借鉴点：使用真实项目和真实任务，而不是只用玩具样例。

在本项目中对应为：

- 样例项目应逐步加入真实开源项目或真实业务项目快照。
- 每个 case 固定版本，避免评估对象漂移。
- 重点评估 skill 在复杂、混乱、不完整项目中的表现。

### 2.7 AgentBench

借鉴点：评估 agent 时不仅看最终答案，也看过程、工具使用和失败类型。

在本项目中对应为：

- 不只评估最终文档，还评估是否读取了正确文件、是否记录 `_facts.md`、是否执行交叉审查。
- 记录失败类型，例如类型错配、过度推断、证据缺失、文档空洞、链接损坏。

## 3. 核心闭环

```text
样例项目集
  -> 调用 project-portrait skill
  -> 生成项目画像文档
  -> 自动评估
  -> 失败归因
  -> 生成改进建议
  -> 修改 skill
  -> 回归运行
  -> 对比新旧版本
  -> 通过门禁后合并
```

这个闭环的重点是“可复现”和“可比较”。同一个样例项目、同一个 prompt、同一个 skill 版本，应该能够生成可归档、可评分、可追踪的结果。

## 4. 样例项目集设计

### 4.1 样例分层

样例项目不要只准备一个 demo，建议分为四层：

| 层级 | 目的 | 示例 |
|------|------|------|
| smoke | 验证 skill 基本流程不崩 | 极小 CLI、极小 React app |
| golden | 人工确认过高质量输出的标准样例 | 小型后端服务、组件库、算法库 |
| adversarial | 故意暴露弱点 | 无 README、混合技术栈、目录混乱、文档过期 |
| real-world | 贴近真实使用 | 开源项目快照、历史业务项目快照 |

### 4.2 类型覆盖

至少覆盖以下项目类型：

- 前端：React/Vue SPA、组件库。
- 后端：FastAPI/Express/Spring 服务。
- 测试：pytest/Jest 测试框架。
- 算法：Python 算法库、ML pipeline。
- SDK：TypeScript/Python package。
- CLI：命令行工具。
- 数据/基础设施：ETL、Terraform、GitHub Actions。
- Monorepo：前后端混合、多 packages 项目。

### 4.3 Eval Case 配置

每个样例项目配一个 `eval_case.yaml`。

```yaml
id: frontend-react-dashboard
type: frontend
goal: onboarding
prompt: "使用 project-portrait 梳理这个项目，输出到 docs_project_portrait"
expected:
  must_have:
    - 读者导览
    - 组件结构
    - 路由/页面流
    - 状态管理说明
  must_not_have:
    - 数据库表设计
    - 后端异步任务调度
  quality_focus:
    - 类型适配
    - 证据追溯
    - 可读性
```

配置文件不应该泄露给生成文档的 agent。它只用于评估阶段，避免生成过程直接迎合评分规则。

## 5. 评估指标体系

评估指标分为三类：确定性指标、半结构化指标、LLM-as-judge 指标。

### 5.1 确定性指标

这些指标应优先用脚本检查：

- 文档是否生成完整。
- 是否存在空标题。
- 是否存在 `[链接]()` 之类的空链接。
- 相对链接是否有效。
- 是否引用不存在的文件路径。
- 是否残留模板占位符。
- 是否出现明显不适用章节，例如前端项目写数据库表设计。
- 是否生成并使用 `_facts.md`。

### 5.2 半结构化指标

这些指标可以用规则 + 简单文本解析检查：

- 每份文档是否有“读者导览”。
- 关键结论是否有证据来源。
- 核心结论是否标注为源码确认、文档确认、推断或待确认。
- `05` 高亮是否回答“如果不这样做会怎样”。
- `06` 优化建议是否绑定具体证据。
- 是否存在泛泛而谈的表述。

### 5.3 LLM-as-judge 指标

这些指标适合由独立 judge 模型辅助评估：

- 类型适配度。
- 事实严谨性。
- 文档可读性。
- 信息密度。
- 面向目标读者的有效性。
- 是否存在过度推断。
- 是否存在模板腔。

### 5.4 建议权重

```text
事实正确性        30%
类型适配          20%
证据可追溯        15%
文档可读性        15%
结构完整性        10%
改进建议质量      10%
```

权重可以随阶段调整。早期更重视“不要错”，后期再提升“写得好”。

## 6. Eval 运行器设计

建议新增独立目录：

```text
evals/
  cases/
    frontend-react-dashboard.yaml
    backend-fastapi-service.yaml
    pytest-plugin.yaml
  sample-projects/
  golden/
  runs/
  scripts/
    run_eval.py
    score_docs.py
    compare_runs.py
    summarize_failures.py
```

一次运行产出：

```text
evals/runs/2026-06-29T210000/
  outputs/
  scores.json
  failures.json
  judge_report.md
  diff_vs_baseline.md
  reflection.md
```

### 6.1 `run_eval.py`

职责：

- 读取 eval case。
- 准备样例项目工作区。
- 调用 skill 生成文档。
- 记录 skill 版本、prompt、时间、样例项目版本。
- 调用评分脚本。

### 6.2 `score_docs.py`

职责：

- 执行确定性检查。
- 执行半结构化检查。
- 调用 LLM judge。
- 输出 `scores.json` 和 `failures.json`。

### 6.3 `compare_runs.py`

职责：

- 对比当前 run 和 baseline run。
- 标记提升、退化和新增失败。
- 生成 `diff_vs_baseline.md`。

### 6.4 `summarize_failures.py`

职责：

- 汇总失败类型。
- 生成 `reflection.md`。
- 给出最小修改建议。

## 7. 失败归因机制

`reflection.md` 是自评估体系的核心产物之一。它不只是列分数，而是解释 skill 为什么失败。

示例：

```md
## 失败归因

- 前端项目中生成了“数据库设计”章节，说明类型适配规则仍然不够强。
- 05 高亮有 2 条无法回答“如果不这样做会怎样”，说明高亮筛选门槛执行不足。
- 06 优化建议出现“增加缓存”这类泛泛建议，说明改进建议需要强制绑定证据。
```

每条失败最好映射到一个可修改对象：

| 失败现象 | 归因 | 修改对象 |
|----------|------|----------|
| 前端项目出现数据库章节 | 类型适配规则不足 | `references/project-types.md` |
| 高亮无法说明反事实价值 | 高亮质量门槛不足 | `SKILL.md` 或质量规则文档 |
| 优化建议泛泛而谈 | 缺少证据绑定要求 | `SKILL.md`、`_facts-template.md` |
| 引用不存在文件 | 缺少链接/路径检查 | `scripts/link_check.py` |

## 8. Skill 升级策略

每次升级应该采用最小补丁原则：

```text
失败 -> 归因 -> 规则缺口 -> 修改位置 -> 预期改善指标 -> 回归验证
```

示例：

```md
问题：前端项目输出了数据库章节。
归因：SKILL.md 虽然要求类型适配，但没有在 03 生成阶段强制删除不适用章节。
修改：在 SKILL.md Phase 3 增加“适用性裁剪规则”。
验证：重新运行 frontend-react-dashboard 和 fullstack-mixed 两个 case。
```

禁止根据单个 case 大幅重写整个 skill。除非多个 case 指向同一类系统性问题，否则优先做局部修正。

## 9. 回归门禁

每次修改 skill 后都要跑回归。

### 9.1 合并条件

- 新版本总分不低于旧版本。
- blocking 问题不能增加。
- 关键 golden case 不能退化。
- 事实错误不能增加。
- 类型错配不能增加。
- 空标题、断链、占位符必须为 0。

### 9.2 问题级别

| 级别 | 含义 | 示例 |
|------|------|------|
| blocking | 必须修复，否则不能合并 | 断链、引用不存在文件、明显类型错配、虚构事实 |
| warning | 需要判断，可延期 | 表达啰嗦、证据粒度不足、章节顺序一般 |
| info | 可优化项 | 局部措辞、可增加图示、可补充阅读路径 |

## 10. 防止自评估失真

自评估体系容易出现“自己奖励自己”的问题，必须加入约束。

建议规则：

- 样例项目固定版本，不能每次变化。
- 保留一组 hidden cases，不用于日常调 prompt。
- LLM judge 的评分 prompt 版本化。
- 确定性检查优先于 LLM judge。
- 定期人工抽检高分 case 和失败 case。
- 生成文档的 agent 不能看到 eval case 的 expected 细节。
- 修改 skill 的 agent 只能看到失败报告，不能看到 hidden case 期望答案。

## 11. 能力分级体系

### L0：人工试用

手动找几个项目运行 skill，人工观察输出质量。

### L1：确定性检查

自动检查：

- 文档是否存在。
- 链接是否有效。
- 是否有空标题。
- 是否有占位符。
- 是否引用不存在文件。

### L2：标准样例评分

每类项目有固定 eval case，结合规则评分和 LLM judge 评分。

### L3：失败归因闭环

自动生成 failure report、reflection 和改进建议，但是否修改 skill 仍由人工决定。

### L4：半自动升级

agent 根据失败报告生成 skill patch，运行回归，通过门禁后提交 PR。

### L5：持续知识维护

不只评估首次生成文档，还评估增量更新能力：

- 给定 git diff，检查项目画像是否正确更新。
- PR 前检查文档是否过期。
- 根据代码变化定位受影响文档。

## 12. 建议落地顺序

### 第一阶段：最小闭环

```text
3 个样例项目
+ run_eval.py
+ 确定性检查
+ 简单 LLM judge
+ scores.json
+ failure_report.md
```

目标：先跑通流程，而不是追求指标完美。

### 第二阶段：类型覆盖

```text
覆盖 8 类项目
+ golden baseline
+ compare_runs.py
+ 失败归因分类
+ 回归门禁
```

目标：验证 skill 是否真的通用，而不是只对某一类项目有效。

### 第三阶段：半自动优化

```text
自动生成改进建议
+ skill patch proposal
+ hidden cases
+ 人工抽检流程
```

目标：让每次升级都有证据、有归因、有回归验证。

## 13. 最终形态

长期目标是把 `project-portrait` 从“静态画像生成器”升级为“项目知识初始化与维护工具”。

最终能力包括：

- 首次运行生成完整项目画像。
- 根据 git diff 增量更新文档。
- PR 前自动检查文档是否过期。
- 为新人生成阅读路线。
- 为面试生成口头讲解稿。
- 为架构评审生成风险清单。

这套体系的核心判断标准是：每次修改都能说明为什么改、改善了什么、有没有回归。
