# 01 - 文档质量提升规则

本文档定义 `project-portrait` skill 生成项目画像文档时的质量标准。目标是让输出从“填模板”变成“有证据、有判断、可阅读、可复查”的项目文档。

## 1. 总体原则

所有交付文档都必须满足五条基本规则：

1. 每节采用“结论 -> 证据 -> 影响”的表达结构。
2. 每份文档开头都有“读者导览”。
3. 每个核心结论必须追溯到源码、文档、推断或待确认。
4. 明确禁止空标题、模板腔和泛泛优化建议。
5. 每个高亮内容必须回答“如果不这样做会怎样”。

这些规则优先级高于固定模板。模板只是结构参考，不能为了填满模板而牺牲准确性和可读性。

## 2. 每节采用“结论 -> 证据 -> 影响”

### 2.1 规则说明

每个重要段落或小节都应该先给结论，再给证据，最后说明影响。

- **结论**：这一节想告诉读者的判断是什么。
- **证据**：这个判断来自哪些源码、配置、测试、文档或推断。
- **影响**：这个事实对理解、维护、扩展、排查或面试表达有什么意义。

### 2.2 推荐结构

```md
### 模块名或设计点

结论：这个模块负责统一处理用户输入校验，是请求进入业务逻辑前的第一道边界。

证据：
- `src/middleware/validation.ts` 中注册了全局校验中间件。
- `src/schemas/user.ts` 定义了用户相关请求的 schema。
- `tests/validation.test.ts` 覆盖了非法参数和缺失字段场景。

影响：后续排查参数错误时，应优先查看 schema 和中间件注册顺序，而不是直接进入 service 层。
```

### 2.3 不推荐写法

```md
### 输入校验模块

这个模块比较重要，主要负责一些校验逻辑，保证系统更稳定。
```

问题：

- “比较重要”没有说明为什么重要。
- “一些校验逻辑”过于模糊。
- 没有证据来源。
- 没有告诉读者如何使用这条信息。

## 3. 每份文档开头必须有读者导览

### 3.1 规则说明

每份文档开头必须用一个短块说明：

- 这份文档回答什么问题。
- 适合谁读。
- 建议先读哪些章节。
- 哪些内容不在本文范围内。

### 3.2 推荐模板

```md
> **读者导览**
> 本文回答“系统如何设计，以及关键边界在哪里”。
> 适合新接手项目的开发者、架构评审者和准备面试复盘的人阅读。
> 如果只想快速理解整体结构，先读第 1、2、4 节。
> 本文不展开每个函数的实现细节，函数级说明见 `04-module-breakdown.md`。
```

### 3.3 适配规则

- 面试模式：导览应告诉读者哪些章节适合转化成口头表达。
- 新人上手模式：导览应告诉读者先看哪些入口、流程和模块。
- 架构评审模式：导览应突出边界、权衡、风险和待确认问题。
- 深度文档模式：导览应说明文档之间的阅读顺序。

## 4. 核心结论必须可追溯

### 4.1 证据等级

每个核心结论都必须属于以下四类之一：

| 证据等级 | 含义 | 示例 |
|----------|------|------|
| 源码确认 | 直接由源码、配置、测试、脚本确认 | 路由定义、组件引用、函数调用、配置默认值 |
| 文档确认 | 由 README、设计文档、注释、issue、commit 信息确认 | 项目目标、业务背景、架构意图 |
| 推断 | 根据结构、命名、依赖关系推断 | 通过表名推断领域对象，通过依赖推断运行环境 |
| 待确认 | 当前仓库证据不足，不能可靠判断 | 用户规模、真实业务流程、历史设计动机 |

### 4.2 写作要求

- 源码确认的结论必须给出文件路径，必要时给出类名、函数名或配置项名。
- 文档确认的结论必须说明来自哪个文档或注释。
- 推断类结论必须写明推断依据，不能伪装成事实。
- 待确认内容必须集中列入“待确认问题”或“风险与未知”章节。

### 4.3 示例

推荐：

```md
系统采用前后端分离结构。这个结论由 `apps/web/package.json`、`apps/api/package.json` 以及 `docker-compose.yml` 中独立的 web/api 服务定义共同确认。
```

推荐：

```md
项目可能面向内部运营人员使用。依据是页面命名集中在 `dashboard`、`admin`、`report`，但仓库没有产品说明文档，因此该用户角色仍需维护者确认。
```

不推荐：

```md
这是一个面向企业客户的运营平台。
```

问题：如果仓库没有明确证据，这句话就是未经验证的业务判断。

## 5. 禁止空标题、模板腔和泛泛优化建议

### 5.1 空标题

禁止保留没有实质内容的标题，例如：

```md
## 数据库设计

暂无。
```

处理方式：

- 如果项目没有数据库，删除该章节。
- 如果该关注点存在但当前证据不足，改为“待确认：数据持久化方式”。
- 如果只是信息少，用一段明确说明替代空标题。

### 5.2 模板腔

禁止出现只是在重复模板、没有项目特异性的表达，例如：

- “该模块提高了系统稳定性。”
- “该设计提升了可维护性。”
- “后续可以优化性能。”
- “系统采用模块化设计。”

这些话只有在回答以下问题后才可以保留：

- 哪个模块？
- 哪个设计？
- 稳定性或可维护性体现在哪里？
- 有什么证据？
- 对读者有什么影响？

### 5.3 泛泛优化建议

优化建议必须绑定当前项目证据。

不推荐：

```md
建议增加缓存，提高系统性能。
```

推荐：

```md
当前 `src/services/report.ts` 每次生成报表都会重新读取完整配置和历史数据，且没有看到缓存或增量计算逻辑。可以优先评估报表输入数据的缓存策略，预期价值是减少重复计算；风险是需要明确缓存失效条件。
```

## 6. 高亮内容必须回答“如果不这样做会怎样”

### 6.1 规则说明

技术高亮不是“用了某个常见技术”，而是“做出了有价值的设计选择”。每个高亮都必须解释如果没有这个设计，系统会出现什么问题。

### 6.2 推荐结构

```md
### 高亮：统一的命令执行上下文

结论：CLI 将参数、配置文件和环境变量合并为统一的执行上下文，再传给各个 command handler。

证据：
- `src/config/resolveConfig.ts` 负责配置合并。
- `src/commands/build.ts` 和 `src/commands/deploy.ts` 都接收同一种 context 对象。

如果不这样做会怎样：每个命令都需要重复解析参数和配置，容易出现不同命令的优先级规则不一致，后续新增命令也更容易引入行为差异。

影响：维护者新增命令时，应优先复用 context 构建逻辑，而不是在 command 内部重新读取环境变量。
```

### 6.3 降级规则

如果一个候选高亮无法回答“如果不这样做会怎样”，则按以下方式处理：

1. 降级为普通模块说明。
2. 合并到相关流程或架构章节。
3. 删除，不强行凑数量。

## 7. 交叉审查清单

生成文档后，逐项检查：

- [ ] 每份文档开头都有读者导览。
- [ ] 每个核心章节都有清晰的结论。
- [ ] 每个核心结论都有证据来源或明确标注为推断/待确认。
- [ ] 每个章节都说明了该事实对读者的影响。
- [ ] 没有空标题。
- [ ] 没有未替换的模板占位符。
- [ ] 没有“提升稳定性”“提高可维护性”这类无证据表述。
- [ ] 每条优化建议都绑定了当前项目的具体证据。
- [ ] 每个高亮都回答了“如果不这样做会怎样”。
- [ ] 无法确认的内容集中列入“待确认问题”或“风险与未知”。

## 8. 与 skill 的集成建议

后续可以把这些规则集成到 `project-portrait` skill 的质量门槛中：

- 在 `SKILL.md` 的质量阶段引用本文档。
- 在 `_facts-template.md` 中增加“结论 -> 证据 -> 影响”的记录字段。
- 在交叉审查阶段增加文档质量检查项。
- 如果后续增加 `claim_scan` 或 `link_check` 脚本，可以把检查结果和本文档的人工审查清单结合使用。
